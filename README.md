# BookAI

Spring Boot приложение для автоматизации работы барбершопа с клиентами на базе AI.

## Возможности

- Общение с клиентами через AI-ассистента
- Ответы на вопросы об услугах, ценах и записи

## Технологии

- Java 25, Spring Boot 4.0.3
- Anthropic Claude API

## Фиксация статуса проекта

### Task 1–6: Базовая инфраструктура

| Задача | Описание |
|--------|----------|
| Task 1 | Инициализация Spring Boot проекта, базовая структура пакетов |
| Task 2 | Интеграция с Anthropic Claude API (`AnthropicClient`) |
| Task 3 | REST-эндпоинт `/api/book` — простой LLM-запрос и ответ |
| Task 4 | Параметр `temperature` для управления креативностью ответов |
| Task 5 | Выбор версии модели (Haiku / Sonnet / Opus) |
| Task 6 | Агент: базовая multi-turn сессия в памяти (`BookAgent`) |

### Task 7: Сохранение контекста агента

**Задача:** агент должен помнить историю диалога после перезапуска приложения.

### Анализ: насколько текущий агент соответствует каноническому определению

> Агент — это функция от 6 параметров. LLM — это класс или интерфейс, агент — конкретная реализация с суженным контекстом под задачу.
> — Алексей Гладков

| Параметр | Что должно быть | Статус |
|----------|-----------------|--------|
| **LLM / Модель** | Мозг агента | ✅ `AnthropicClient`, выбор модели через UI |
| **System prompt** | Личность, роль и ограничения агента | ❌ Передаётся `null` — модель не знает кто она |
| **History** | История текущего диалога | ✅ `AgentSessionStore` + H2, переживает рестарт |
| **Memory** | Долгосрочная память о пользователе между сессиями | ⚠️ Есть история, но нет отдельного слоя памяти |
| **Tools** | Инструменты: поиск, калькулятор, внешние API | ❌ Отсутствуют |
| **Input / Output** | Структурированный вход и выход | ✅ `AgentChatRequest` / `AgentChatResponse` |

**Вывод:** текущий агент — это продвинутый stateful чат-бот. Настоящий агент отличается наличием agentic loop: LLM сама решает, какой инструмент вызвать, получает результат и принимает следующее решение. Этого цикла пока нет.

**Что нужно для полноценного агента:**
- `System prompt` — задать роль: _«Ты помощник барбершопа, помогаешь с записью и вопросами об услугах»_
- `Tools` — например, проверка доступных слотов, запись клиента, получение прайса
- `Agentic loop` — цикл: вызов LLM → решение об инструменте → выполнение → снова LLM

### Task 8: Работа с токенами

**Задача:** добавить подсчёт токенов для текущего запроса, всей истории диалога и ответа модели. Показать как растут токены и что происходит при переполнении.

**Что реализовано:**

| Компонент | Изменение |
|-----------|-----------|
| `AgentChatResponse` | Добавлены `inputTokens`, `outputTokens`, `totalInputTokens`, `totalOutputTokens`, `turnNumber` |
| `AgentMessageEntity` | Хранит `inputTokens` / `outputTokens` каждого хода в БД |
| `AgentSessionStore` | Методы накопления токенов по сессии через SQL SUM |
| `AgentBook` | `max-tokens` вынесен в `application.yml`, обработка overflow с сохранением маркера `[ERROR]` в историю |
| `InfoController` | Отдаёт `maxTokens` на фронтенд |

**UI (сайдбар агента):**
- Панель **«Текущий запрос»** — новое сообщение ≈, история (весь контекст), ответ модели
- Панель **«За всю сессию»** — оплачено input/output, стоимость сессии в $ по тарифу выбранной модели
- Бар-чарт роста контекста по ходам
- Планка **«Последний ответ / max_tokens»** — синяя → жёлтая (≥80%) → красная (обрезан)

**Ключевые выводы:**
- `max-tokens` — лимит длины одного ответа, не контекста; при превышении модель обрывает ответ без ошибки
- `inputTokens` каждого запроса включает всю историю — контекст растёт по треугольной прогрессии O(n²)
- Реальный overflow наступает при ~200 000 input-токенов (лимит контекста модели)

### Task 9: Сжатие контекста

**Задача:** реализовать сжатие контекста агента — заменять старую историю саммари, чтобы сдерживать рост input-токенов без потери смысла диалога.

**Результаты теста (30 сообщений, модель Haiku):**

| Метрика | Без сжатия | Со сжатием | Δ |
|---|---|---|---|
| Input-токены (финал) | 16 463 | 2 666 | **-83.8%** |
| Оплачено input (итого) | 179 713 | 62 050 | **-65.5%** |
| Оплачено output (итого) | 16 065 | 24 453 | +52.2% |
| Стоимость сессии, $ | 0.20803 | 0.147452 | **-29.1%** |

**Рост input-токенов по ходам:**

| Ход | Без сжатия | Со сжатием | Экономия |
|-----|-----------|-----------|---------|
| 1   | 21        | 0         | 100.0%  |
| 6   | 1 155     | 1 303     | -12.8%  |
| 11  | 4 071     | 2 677     | 34.2%   |
| 16  | 7 026     | 2 337     | 66.7%   |
| 21  | 11 432    | 2 675     | 76.6%   |
| 26  | 15 443    | 2 670     | 82.7%   |

**Контекст (финал):** 5 сообщений как есть + 52 в саммари

**Ключевые выводы:**
- После 10–11 хода контекст со сжатием стабилизировался на ~2 600–2 700 токенов, без сжатия рос линейно до 16 463
- Сжатие срабатывает корректно — старая история сворачивается в саммари через отдельный LLM-вызов
- Рост output-токенов (+52%) — нормален: модель генерирует более развёрнутые ответы при полном, но сжатом контексте
- Итоговая экономия стоимости сессии: **-29.1%** при сохранении качества ответов
